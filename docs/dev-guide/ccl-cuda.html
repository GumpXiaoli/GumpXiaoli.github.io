Connected Component Labeling on Device

General Idea​

我们提出了一个新版本的两遍联合查找算法，其中所有迭代都在内核中实现。对于输入数据的给定维度，执行的内核的数量是固定的，并且不需要主机和设备之间的同步。我们的方法还利用GPU上的共享内存，因为我们将输入数据划分为更小的区块，其中CCL问题在本地解决。然后使用树状分层合并方案递归合并切片，如图5所示。在合并方案的每个级别，我们使用相邻区块的边界区域来合并来自较低级别的CCL问题的本地解决方案。

我们算法的所有步骤都完全在GPU上的四个不同内核中实现（见图6）。内核1解决了共享内存中输入数据的小块的CCL问题。然后，将合并方案递归地应用于区块之间的边界区域。合并方案使用两个不同的内核。一个将来自一组区块的等价树合并在一起，并且第二个在合并方案的每次迭代之前更新数据区块的边界区域上的标签。合并方案仅用于合并来自连接区块的等价树;因此，当合并所有级别上的所有区块时，有必要通过更新所有数据元素上的标签来找到CCL问题的最终解决方案。此过程在最后一个内核中完成。第一个内核和最终内核都只调用一次，而合并方案内核则为每个级别的合并方案调用。级别数由输入数据的大小决定;因此，它不依赖于分析数据中连接组件的结构。这是我们算法的一大优势，因为它减少了执行内核的数量，并且还消除了从设备到主机的所有数据传输。尽管对于给定的输入数据维度，内核调用的数量是固定的，但是即使它们具有相同的维度，仍然需要执行CCL求解器的不同迭代次数。但是，所有这些迭代都直接在内核1和内部2中执行，并且它们实际上是从主机隐藏的。内核将在下一节中详细介绍。

Implementation Details​

CCL算法是用CUDA 3.0和C ++编写的，所有源代码都可以在附带的演示应用程序中找到。 CCL算法需要具有1.3或更高计算能力的GPU，因为内核在全局和共享内存上都使用原子操作。 演示应用程序还需要CUDA Toolkit 3.0，因为它利用了此版本中引入的OpenGL互操作性功能。

连接标准必须首先由用户明确定义，并且它是我们算法的参数之一。 例如，我们可以说如果RGB图像的两个元素的亮度之间的差异低于给定的阈值，则它们会被连接起来。 定义连通性标准的另一种方法是对输入数据应用分段。 在分段数据中，两个相邻元素仅在它们共享相同的段值时才连接，这也是我们在实现中选择的方法。 输入数据存储在8位无符号整数的二维数组中，该数组包含所有元素的段值，但表示背景的段值0除外，我们的CCL算法会忽略它们。

然后使用来自Moore邻域的分段值检测每个非背景元素的连通分量。 所有连接的组件都由标签表示，这些标签存储在32位整数2-D数组中，该数组与输入数据具有相同的维度。

内核1首先在GPU的共享内存中找到输入数据的小块的CCL问题的局部解决方案（参见图8中的内核的伪代码）。每个图块由一个CUDA线程块解决，并且每个线程对应于给定图块中的单个元素。本地解决方案是从图3中描述的双通CCL求解器的修改版本中获得的。每当任何两个相邻的连接元素具有不同的标签时，内核就执行修改的CCL算法的新迭代。每次迭代都执行算法的两次传递，但第一次传递分为两个单独的步骤。首先，每个线程检测来自所有相邻元素的最低标签。如果相邻元素的标签低于处理元素的标签（两个连接元素属于不同的等价树），那么我们使用atomicMin函数合并这些等价树。原子操作是必要的，因为同一个等价树可以同时由多个线程更新。该CCL算法在图7中的一个简单示例中进行了演示。最终标签与共享内存的地址空间相关，因为它为内存访问提供了更简单的寻址模式。因此，在所有迭代完成后，必须将标签传输到全局地址空间，然后将它们存储在全局设备存储器中。

要计算CCL问题的全局解决方案，有必要合并上一步骤中的本地解决方案。我们可以使用两个图块的所有边界元素之间的连接来合并相应图块中的连接组件的等价树。因为我们想要问题的并行解决方案，所以通常需要执行此合并操作的多次迭代，因为我们无法保证在单个合并操作之后合并所有等价树。这种迭代机制需要在单个线程块的各个线程之间进行同步，但一般来说，处理一个线程块中的所有边界元素是不可行的，因为既没有足够的线程，也没有足够的共享内存来存储所有需要的数据。我们对此问题的解决方案在内核2中实现，如图9所示。给定的一组图块合并在一个3D线程块中，其中块的x和y维度用于索引单个图块，而z维度包含用于比较和合并给定图块边界上的等价树的单独线程使用Union功能。因为与每个图块相关联的线程数量不足以处理给定图块的所有边界元素，所以所有线程实际上顺序地处理多个边界元素。在第一阶段，线程处理底部水平边界两侧的元素，而在第二阶段，它们处理右侧垂直边界的两侧。如果两个图块之间的边界太长，则线程会顺序处理一个边界，然后再继续下一个边框。正如我们在图9中看到的，四个切片之间的角元素由多个线程处理，但这些冗余线程的数量很少，不需要特殊处理。

内核2的一次执行可以仅将有限数量的区块合并在一起，因此有必要多次调用内核，其中内核的每次迭代将来自先前步骤的合并区块作为输入。每个级别合并的切片数量可能会有所不同。我们发现合并4×4区块会产生很好的效果。如果我们在一个级别上合并更多的图块，那么与每个图块相关联的线程数量太小，整个过程变得更加顺序。另一方面，如果我们合并较少的区块，则合并方案的级别数量增加 - 这也是不可取的结果。内核2的伪代码如图10所示。使用Union函数合并来自相邻区块的连接组件（等价树）。 Union功能的实现遵循图4的描述。但是，多个线程可能会尝试同时合并两个等价树，这就是使用CUDA atomicMin函数实现合并操作的原因。当两个连接元素位于不同的等价树（具有不同根标签的树）时，我们将共享变量sChanged设置为1，这表示该算法的另一次迭代是必要的。

值得一提的是，内核2的输出并不是给定区块的CCL问题的完整解决方案，因为它只将不同的等价树合并在一起，但非根元素上的标签保持不变。但是，可以保证合并区块中的每个等价树代表最大可能的连接组件。这种方法的优点是标签仅在相对较少数量的元素上更新，并且写入全局存储器的数据量通常很小。缺点是，当我们将越来越多的图块合并在一起时，合并图块的边框元素上的等价树的深度可能会显着增加，从而使联合操作变慢。为了解决这个问题，我们使用内核3使用FindRoot操作在区块之间的边框元素上展平等价树（参见图4）。内核总是在两个连续的内核2调用之间执行，如图6所示。虽然我们的方法即使没有内核3也可以工作，但使用它的观察到的性能提升大约是15-20％。
